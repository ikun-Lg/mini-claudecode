# 生成测试代码

请帮我为指定的代码生成测试。生成测试时遵循以下规范：

## 测试原则

1. **测试行为而非实现**：测试应该验证代码的外部行为，而非内部实现细节。这样重构时测试不需要频繁修改。
2. **Arrange-Act-Assert 模式**：每个测试用例按三段式组织：
   - **Arrange** — 准备测试数据和前置条件
   - **Act** — 执行被测试的代码
   - **Assert** — 验证结果
3. **单一职责**：每个测试用例只验证一个行为点，一个 assert 聚焦一个期望。
4. **命名清晰**：测试名应描述被测行为和预期结果，如 `should return empty array when input is null`。

## 测试用例覆盖

| 类别     | 说明                                         |
| -------- | -------------------------------------------- |
| 正常路径 | 典型输入下的正确输出                         |
| 边界值   | 空值、零值、最大/最小值、单元素               |
| 异常输入 | 非法类型、格式错误、越界值                   |
| 错误处理 | 验证错误是否被正确抛出或处理                 |
| 副作用   | 验证函数对外部状态的影响（如写文件、发请求） |

## 执行步骤

1. 用 `read_file` 阅读被测代码，理解其功能和接口
2. 检查项目是否已有测试框架（查看 `package.json` 的 devDependencies），优先使用项目已有的框架
3. 根据代码路径推断测试文件位置（如 `src/utils.js` → `test/utils.test.js` 或 `__tests__/utils.test.js`）
4. 生成测试代码，覆盖正常路径、边界值和异常输入
5. 用 `confirm` 工具确认后写入测试文件
6. 用 `bash` 工具运行测试，验证是否通过

## 常见测试框架

- **Jest** — `describe` / `it` / `expect`
- **Vitest** — `describe` / `it` / `expect`（API 与 Jest 兼容）
- **Mocha + Chai** — `describe` / `it` / `expect`
- **Node.js 内置** — `node:test` 模块（Node 18+）

## Mock 规范

- 优先 Mock 外部依赖（网络请求、文件系统、数据库），而非内部逻辑
- Mock 应尽量贴近真实行为，避免 Mock 导致测试失去意义
- 在测试 teardown 中恢复所有 Mock，避免影响其他测试
