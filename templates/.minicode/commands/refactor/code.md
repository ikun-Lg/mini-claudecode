# 代码重构

请帮我重构指定的代码。重构时遵循以下原则：

## 重构原则

1. **不改变外部行为**：重构后代码的功能行为必须与重构前完全一致，不引入新功能，不改变接口。
2. **小步推进**：每次只做一类重构，完成后验证，再进行下一步。避免一次性大改导致难以定位问题。
3. **保持测试通过**：如果项目有测试，每步重构后确保测试仍然通过。如果没有测试，先建议补充关键路径的测试。

## 常见重构手法

| 手法               | 适用场景                                   |
| ------------------ | ------------------------------------------ |
| 提取函数           | 函数过长，部分逻辑可独立                   |
| 内联函数           | 函数过于简单，间接层增加理解成本           |
| 提取变量           | 复杂表达式难以理解                         |
| 重命名             | 命名不准确，无法表达意图                   |
| 移动函数/字段      | 函数或字段放错了类/模块                    |
| 用多态替换条件     | 复杂的条件分支逻辑                         |
| 用策略模式替换开关 | 多种算法/行为切换                          |
| 引入参数对象       | 多个函数共享一组参数                       |
| 以查询替换临时变量 | 临时变量只被赋值一次且用于简化表达式       |

## 执行步骤

1. 先用 `read_file` 阅读目标代码，理解现有逻辑
2. 分析代码中的「坏味道」（bad smells），识别可重构的点
3. 用 `confirm` 工具向用户说明重构计划，等待确认
4. 逐步执行重构，每步使用 `edit_file` 修改代码
5. 重构完成后，回顾整体代码，确认无遗漏
6. 如果有测试，运行测试验证

## 注意事项

- 重构前确保代码当前可正常工作
- 不要在重构的同时修复 Bug 或添加功能
- 如果代码缺乏测试覆盖，优先建议补充测试再重构
