# 安全规范

当编写涉及安全相关的代码或配置时，请严格遵循以下规范：

## 敏感信息保护

1. **禁止硬编码密钥**：不得在代码中硬编码 API Key、Token、密码、连接字符串等敏感信息。统一使用环境变量或配置文件管理。
2. **配置文件示例**：提供配置示例时使用占位符，如 `API_KEY=your_api_key_here`，不使用真实值。
3. **`.env` 文件**：`.env` 文件必须加入 `.gitignore`，不得提交到版本控制。提供 `.env.example` 作为模板。

## 输入校验

1. **不信任外部输入**：所有用户输入、API 请求参数、命令行参数必须经过校验后再使用。
2. **SQL 注入防护**：使用参数化查询或 ORM，禁止拼接 SQL 字符串。
3. **XSS 防护**：对输出到 HTML 的内容进行转义，使用框架自带的转义机制（如 React 的 `{}` 自动转义）。
4. **路径穿越防护**：处理文件路径时校验 `..` 等路径穿越字符，确保路径在允许范围内。

## 依赖安全

1. **锁版本**：使用 `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` 锁定依赖版本。
2. **审计依赖**：定期运行 `npm audit` 或 `yarn audit` 检查已知漏洞。
3. **最小依赖**：只引入必要的依赖，避免引入庞大的库来解决小问题。

## 权限与认证

1. **最小权限原则**：生成的代码只请求所需的最小权限。
2. **认证校验**：API 接口必须进行身份认证和权限校验，不暴露未授权的端点。
3. **CORS 配置**：不使用 `Access-Control-Allow-Origin: *`，明确指定允许的来源。

## 危险操作

1. **数据库操作**：涉及 `DROP`、`DELETE`、`TRUNCATE`、`UPDATE` 等危险 SQL 时，必须先提示用户确认。
2. **文件删除**：删除文件 / 目录操作前提示用户确认。
3. **批量操作**：批量修改或删除数据时，先在小范围验证，再执行完整操作。
4. **部署发布**：涉及部署、发布、上线等操作时，说明影响范围并请求确认。
