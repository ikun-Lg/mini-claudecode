# 代码风格规范

当编写或修改代码时，请严格遵循以下规范：

## 通用原则

1. **一致性优先**：遵循项目现有代码风格，不引入新的风格变体。如果项目已有 ESLint / Prettier / EditorConfig 等配置，以配置为准。
2. **简洁可读**：代码以可读性为第一优先级，避免过度炫技。复杂逻辑需添加注释说明意图。
3. **单一职责**：函数 / 类 / 模块只做一件事，函数体尽量控制在 50 行以内。

## 命名规范

- **变量 / 函数**：使用 `camelCase`（JS/TS/Java）或 `snake_case`（Python/Go/Rust），与项目主语言约定一致。
- **常量**：使用 `UPPER_SNAKE_CASE`，如 `MAX_RETRY_COUNT`。
- **类 / 接口 / 类型**：使用 `PascalCase`，如 `UserService`、`HttpResponse`。
- **私有成员**：JS/TS 使用 `#` 前缀或 `_` 前缀，与项目现有风格保持一致。
- **布尔值**：使用 `is`/`has`/`should`/`can` 前缀，如 `isVisible`、`hasPermission`。

## 格式化

- 缩进：2 空格（JS/TS/JSX/TSX/CSS/HTML）或 4 空格（Python/Go），与项目一致。
- 行宽：建议不超过 120 字符。
- 引号：JS/TS 使用单引号 `'`，JSX 属性使用双引号 `"`，与项目现有风格一致。
- 分号：遵循项目现有风格（加或不加），不混用。
- 尾逗号：多行对象 / 数组最后一项加尾逗号，便于 diff 和增删。

## 注释

- 使用 `//` 单行注释，`/** */` 多行注释。
- 注释说明「为什么」而非「做了什么」，代码本身应能表达做了什么。
- 函数注释使用 JSDoc / TSDoc 格式，标注参数类型和返回值。
- TODO 注释格式：`// TODO(作者): 描述`，如 `// TODO(luoguang): 需要处理边界情况`。

## 异常处理

- 不要吞掉异常（空 catch），至少要 `console.error` 或记录日志。
- 自定义错误类继承自 `Error`，包含有意义的错误信息和错误码。
- 错误消息使用中文描述，便于排查。
