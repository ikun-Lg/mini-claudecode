# 代码审查技能

当用户要求进行代码审查（Code Review）时，严格按本技能的流程执行。

## 审查流程

### 第一步：理解上下文
1. 阅读被审查的代码文件，理解其功能和所在模块的作用
2. 检查是否有对应的测试文件，了解测试覆盖情况
3. 查看相关的依赖文件（如 package.json、import 语句），理解技术栈

### 第二步：逐项检查

#### 🔴 严重问题（必须修复）
- [ ] **安全漏洞**：SQL 注入、XSS、命令注入、硬编码密钥
- [ ] **数据丢失风险**：未处理的异步错误、数据库操作无事务
- [ ] **逻辑错误**：条件判断反转、死代码、无限循环
- [ ] **权限问题**：未授权访问、越权操作

#### 🟡 重要问题（推荐修复）
- [ ] **错误处理**：空 catch、未处理的 Promise rejection、错误吞没
- [ ] **边界条件**：空数组、null/undefined、零值、越界索引
- [ ] **资源泄漏**：未关闭的文件句柄、数据库连接、定时器
- [ ] **竞态条件**：异步操作共享状态、定时器回调时机
- [ ] **性能问题**：N+1 查询、O(n²) 循环、不必要的重复计算

#### 🟢 一般问题（建议改进）
- [ ] **命名**：变量/函数名是否清晰表达意图
- [ ] **函数长度**：是否过长（>50行），是否职责单一
- [ ] **重复代码**：是否有可抽取的公共逻辑
- [ ] **注释**：是否缺少必要注释，注释是否有价值
- [ ] **代码风格**：是否与项目现有风格一致

### 第三步：输出审查报告

按以下格式输出：

```
## 审查总结
（整体评价：1-2 句话）

## 🔴 严重问题
1. [文件名:行号] 问题描述
   修复建议：...

## 🟡 重要问题
1. [文件名:行号] 问题描述
   改进建议：...

## 🟢 一般问题
1. [文件名:行号] 问题描述

## ✅ 亮点
（值得肯定的设计或实现）
```

## 审查原则

1. **对事不对人**：审查代码而非审查人，用「这段代码」而非「你的代码」
2. **给出具体建议**：不只指出问题，还要给出可操作的修复方案
3. **区分优先级**：严重问题必须修复，一般问题可视情况处理
4. **肯定亮点**：发现好的设计或实现时，也要点出来
