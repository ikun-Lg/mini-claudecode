# 代码重构技能

当用户要求重构代码时，严格按本技能执行，确保重构不改变代码的外部行为。

## 前置检查

1. **代码可正常工作**：重构前确认代码当前无 Bug。如果有 Bug，先修 Bug 再重构。
2. **有测试覆盖**：检查是否有测试。如果有，重构后运行测试验证。如果没有，先建议补充关键路径测试。
3. **版本可控**：确认代码在版本控制下，重构前确保工作区干净（`git status`）。

## 重构流程

### 1. 识别坏味道（Bad Smells）
阅读代码，识别以下常见问题：

| 坏味道             | 表现                                   | 推荐手法             |
| ------------------ | -------------------------------------- | -------------------- |
| 过长函数           | 函数超过 50 行                         | 提取函数             |
| 过长参数列表       | 参数超过 4 个                          | 引入参数对象         |
| 重复代码           | 多处相似代码                           | 提取函数/模板方法    |
| 发散性变化         | 一个类因不同原因被修改                 | 拆分类               |
| 霰弹式修改         | 一个修改要改多个类                     | 移动函数/合并类      |
| 特性依恋           | 函数大量使用其他类的数据               | 移动函数             |
| 数据泥团           | 多处出现相同的参数组                   | 提取对象             |
| 基本类型偏执       | 用基本类型代替小对象                   | 引入值对象           |
| switch 语句        | 复杂的 switch/if-else                  | 多态替换/策略模式    |
| 平行继承体系       | 修改一个类要同时修改另一个             | 移动字段/函数        |
| 冗余类             | 类的职责太少                           | 内联类               |
| 夸夸其谈           | 过度设计、不必要的抽象                 | 内联函数/内联类      |
| 临时字段           | 字段只在特定情况下有值                 | 提取类               |
| 消息链             | `a.getB().getC().getD()`              | 隐藏委托             |
| 中间人             | 类大量方法只是委托给其他类             | 移除中间人/内联      |
| 不当依赖           | 依赖了不需要的模块                     | 最小化依赖           |

### 2. 制定计划
列出重构步骤，每步只做一种重构。用 `confirm` 工具向用户确认计划。

### 3. 逐步执行
每一步：
1. 用 `edit_file` 修改代码
2. 如果有测试，运行测试确认通过
3. 确认无误后进行下一步

### 4. 最终验证
- 运行全部测试
- 用 `read_file` 通读重构后的代码，确认逻辑一致
- 对比重构前后的核心逻辑，确保行为不变

## 重构原则

1. **小步快跑**：每步重构都很小，随时可以停下
2. **测试驱动**：有测试就频繁运行，没测试就格外小心
3. **不改接口**：重构不改变对外接口，避免影响调用方
4. **不留半成品**：如果重构中途被打断，要说明哪些已完成、哪些未完成
