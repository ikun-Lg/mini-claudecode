# mini-claudecode Roadmap

> 目标：从「能跑的 Demo」演进到「日常工作中可稳定使用」的终端 AI 编程助手。
>
> 优先级定义：
> - **P0 关键**：不解决会在真实工作中「翻车」，属于阻塞项
> - **P1 高**：解决后可用性和体验显著提升
> - **P2 中**：完善功能、补齐对标 Claude Code / CatPaw 的能力
> - **P3 低**：长期演进方向，非必需但有价值

---

## 一、现状评估

### 已具备的能力

| 模块 | 状态 | 说明 |
| --- | --- | --- |
| 流式对话 | ✅ | OpenAI 兼容 API，逐字流式输出 |
| 工具调用循环 | ✅ | 递归执行 `tool_calls`，16+ 内置工具 |
| MCP 协议 | ✅ | stdio / SSE / streamableHTTP 三种传输 |
| 斜杠指令 | ✅ | 内置 + 自定义，阻断 / 非阻断两类 |
| 技能系统 | ✅ | frontmatter 头部注入 + skill 工具加载详情 |
| 规则匹配 | ✅ | 按 `paths` glob 匹配文件引用 |
| 记忆系统 | ✅ | 项目级 + 用户级，`/memory` 自动生成 |
| RAG 知识库 | ✅ | LanceDB + 本地 embedding 降级 |
| 文件 / 图片引用 | ✅ | `@文件` / `#图片` |
| 历史对话 | ✅ | 退出保存 + `/history` 切换 |
| 终端 UI | ✅ | React + Ink，Markdown 渲染 |
| 前端调试 | ✅ | `debugger_page` + `diff_pic` |

### 核心差距（对比可工作水平）

1. **LLM 交互不健壮**：无重试、无中断、无 token 管理、无递归深度限制
2. **工具能力有硬伤**：`bash` 无流式输出、`edit_file` 无备份回滚、`grep`/`glob` 同步遍历大项目卡顿
3. **配置不可控**：无法运行时切换模型、无法查看 / 修改设置、API Key 报错不友好
4. **安全边界缺失**：无工作目录限制、无 `.gitignore` 集成、API Key 明文存储
5. **体验细节粗糙**：无 Ctrl+C 中断、无 token 用量、无语法高亮、长对话性能下降
6. **质量保障空白**：零测试、零 CI、无 lint、无类型定义

## 二、P0 — 关键阻塞项（必须完成）

### 2.1 LLM 请求健壮性

| # | 任务 | 涉及文件 | 说明 |
| --- | --- | --- | --- |
| 1 | 工具调用递归深度限制 | `src/request/llm.js` | `getAIResponse` 递归无上限，模型陷入「调用-调用」循环时会无限消耗 token。增加 `maxToolRounds`（默认 20） |
| 2 | 网络重试与限流处理 | `src/request/llm.js` | 429 / 5xx 指数退避重试 3 次；网络错误重试 1 次；流式请求超时 60s |
| 3 | 流式生成中断 | `src/request/llm.js` + `App.jsx` | 支持 Esc / Ctrl+C 中断正在生成的回复，已生成部分保留 |
| 4 | 上下文窗口管理 | `src/request/llm.js` | 对话超过模型 context window 时，自动摘要或截断早期消息（保留首尾） |
| 5 | token 用量统计 | `src/request/llm.js` | 记录每次请求的 prompt / completion tokens，在状态栏展示累计 |

### 2.2 工具安全与可靠性

| # | 任务 | 涉及文件 | 说明 |
| --- | --- | --- | --- |
| 6 | 工作目录限制 | `write_file` / `edit_file` / `multi_edit` | 默认禁止写入 cwd 之外，通过 confirm 显式放行；防止误改系统文件 |
| 7 | `.gitignore` 集成 | `src/tools/local/fileWalk.js` + `fileSearch.js` | 文件遍历和 `@` 引用自动跳过 gitignore 忽略的文件（node_modules 已跳，但 .env / dist 等需补） |
| 8 | `bash` 命令安全 | `src/tools/local/bash.js` | Windows 下 PowerShell 命令拼接存在注入风险，改用 `-Command` 数组传参或转义；长命令支持流式输出（`spawn` 替代 `exec`） |
| 9 | `edit_file` 备份回滚 | `src/tools/local/edit_file.js` | 写入前保存 `.bak`，失败或验证不通过时回滚；`multi_edit` 同理 |

### 2.3 配置与可观测

| # | 任务 | 涉及文件 | 说明 |
| --- | --- | --- | --- |
| 10 | API Key 配置校验 | `src/request/llm.js` + `src/app.js` | 启动时检测 `apiKey` 是否存在，缺失时引导用户编辑配置而非崩溃 |
| 11 | `/settings` 指令 | `src/commands/index.js` | 查看当前合并后的配置（脱敏 apiKey）；支持 `/settings:model <name>` 热切换模型 |
| 12 | 配置热加载 | `src/request/llm.js` | `loadSettings()` 改为每次请求读取或提供 `reloadSettings()`，避免改配置后需重启 |

## 七、快速启动建议

如果要立刻开始让项目「能工作」，建议按以下顺序推进：

### 第一步：止血（1-2 天）

1. **#1 工具调用递归深度限制** — 10 行代码，防止 token 爆炸

3. **#3 流式生成中断** — 让用户能 Esc 停止跑飞的回复
4. **#9 `edit_file` 备份回滚** — 写入失败不丢原文

1. **#6 工作目录限制** — 防止误改系统文件

3. **#8 `bash` 命令安全** — Windows 下命令注入修复

### 第三步：补核心体验（3-5 天）

1. **#16 `bash` 流式输出** — `npm install` 等长命令不再卡死
2. **#11 `/settings` 指令** — 运行时可控
3. **#5 token 用量统计** — 可观测
4. **#27 RAG `chunkSize` 调优** — 一行配置改动，检索质量提升

### 第四步：按里程碑推进

参考第六节里程碑表，从 v1.1 逐步推进到 v2.0。

## 八、验收标准

每个里程碑完成后，应满足以下「能工作」的验收标准：

### v1.1 验收

- [ ] 在 10k 文件的真实项目上跑 30 分钟不崩溃
- [ ] LLM 请求失败时自动重试，最终失败有友好提示
- [ ] Esc 能中断生成中的回复
- [ ] 编辑文件失败时不丢原文
- [ ] 配置缺失时引导用户而非报栈
- [ ] 无法写入 cwd 之外的文件（除非显式确认）

### v1.5 验收（达到「日常可用」水平）

- [ ] 长对话（100+ 轮）流畅，无明显卡顿
- [ ] `npm install` 等长命令实时输出
- [ ] 代码块有语法高亮
- [ ] 记忆增量更新，不全量覆盖
- [ ] RAG 检索结果相关且质量稳定
- [ ] MCP 工具加载状态可见
- [ ] 前端调试流程（截图 + diff）稳定可用

---

**总结**：项目基础架构已具备（流式、工具循环、MCP、记忆、RAG），核心差距在**健壮性、安全边界、可观测性**三个方面。建议优先完成 P0 的 12 项任务，即可达到「不翻车」的基本可用水平；再推进 P1 即可达到「日常顺手」的工作水平。
